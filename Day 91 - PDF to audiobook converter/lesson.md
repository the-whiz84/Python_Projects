# Day 91 - PDF to Audiobook Converter

This project is a small automation script, but it models a real workflow that shows up in production systems all the time: take a local document, convert it into a machine-usable form, hand it to a cloud service, and let that service produce a new artifact.

In this case, the input is a PDF and the output is speech generated by AWS Polly. The code is short, but the lesson covers several important ideas at once:

- extracting text from a binary document format
- separating local processing from cloud processing
- keeping credentials out of the source code
- using an asynchronous AWS task instead of trying to do everything locally

## 1. Treat the Script as a Pipeline

The easiest way to understand this project is as a three-step pipeline:

1. read the PDF
2. extract plain text
3. send that text to Polly so it can generate an MP3

That mental model matters because the script is not really "about AWS" or "about PDFs" in isolation. It is about connecting distinct tools into one useful automation.

The input file is defined right at the top:

```python
pdf_path = "./Privacy-Policy.pdf"
```

That keeps the workflow explicit. The script is written for one concrete document, but the structure could easily be reused for any other PDF input later.

## 2. Extract Text from the PDF Page by Page

The text extraction lives in its own function:

```python
def pdf_to_text(pdf_path):
    with open(pdf_path, "rb") as file:
        reader = pypdf.PdfReader(file)
        text = ""
        for page_num in range(len(reader.pages)):
            text += reader.pages[page_num].extract_text()
    return text
```

This is a good design choice for two reasons.

First, it keeps the document-processing logic separate from the AWS setup. If something goes wrong while reading the PDF, you know which part of the script to inspect.

Second, it works page by page instead of assuming the whole file can be treated as one opaque object. That is a practical habit in document automation, because PDFs often contain formatting quirks, missing text layers, or pages that extract imperfectly.

Even in this small example, that separation makes the script easier to test mentally:

- Does the PDF open?
- Does text extraction return a string?
- Only after that should Polly be involved.

## 3. Read Cloud Configuration from the Environment

Before the script can talk to AWS, it needs credentials and storage configuration:

```python
aws_access_key_id = os.getenv("AWS_ACCESS_KEY_ID")
aws_secret_access_key = os.getenv("AWS_SECRET_ACCESS_KEY")
aws_s3_bucket = os.getenv("AWS_S3_BUCKET")
region_name = os.getenv("AWS_REGION", "eu-central-1")
```

This is exactly the right pattern for cloud automation.

The source code contains the logic for using AWS, but it does not contain the secret values needed to authenticate with AWS. That keeps the project safer, and it also makes the script portable between environments.

You can run the same code:

- on your machine
- on another developer's machine
- on a CI runner or cloud environment

without changing the source, as long as the environment variables are configured correctly.

## 4. Use Polly as a Managed Text-to-Speech Service

Once the text is extracted, the script creates a Polly client through `boto3`:

```python
polly_client = boto3.Session(
    aws_access_key_id=aws_access_key_id,
    aws_secret_access_key=aws_secret_access_key,
    region_name=region_name,
).client("polly")
```

That line is important because it shows what `boto3` is really doing: it is giving Python a configured client for a specific AWS service. After that, the script can ask Polly to do speech synthesis without handling low-level HTTP details itself.

## 5. Start an Asynchronous Speech Task Instead of Blocking Locally

The final step is where the project becomes more realistic:

```python
response = polly_client.start_speech_synthesis_task(
    VoiceId="Joanna",
    OutputFormat="mp3",
    Text=text,
    Engine="neural",
    OutputS3BucketName=aws_s3_bucket,
)
```

This uses `start_speech_synthesis_task()` rather than a synchronous call that returns audio bytes directly.

That choice matters. It means:

- AWS handles the speech generation remotely
- the script can hand off the work and finish
- the output is written to S3 as a generated artifact

This is a stronger automation pattern than trying to keep everything in memory. For longer documents especially, asynchronous processing is often the better fit.

It also makes the pipeline easier to reason about:

- local Python extracts text
- Polly turns text into audio
- S3 becomes the delivery location for the result

That is a clean separation of responsibilities.

## 6. Why This Project Is More Than a Toy Script

The script is short, but it teaches a pattern you can reuse widely:

- parse a local file
- normalize its contents
- send the clean result to an external service
- retrieve or store the generated output somewhere durable

That is the same broad architecture behind many automation tools:

- document summarizers
- image processing pipelines
- report generation systems
- upload-and-transform workflows

What changes is the service in the middle. The core Python idea stays the same.

## How to Run the PDF-to-Audiobook Script

Set the required AWS environment variables first:

- `AWS_ACCESS_KEY_ID`
- `AWS_SECRET_ACCESS_KEY`
- `AWS_S3_BUCKET`
- optionally `AWS_REGION`

Then install the dependencies and run the script:

```bash
pip install -r requirements.txt
python main.py
```

After running it, verify:

- the PDF is readable by `pypdf`
- the script starts a Polly synthesis task successfully
- the resulting MP3 is written to the configured S3 bucket

## Summary

Day 91 shows how Python can act as the glue between local documents and managed cloud services. You extracted text from a PDF page by page, kept AWS credentials in environment variables, created a Polly client with `boto3`, and started an asynchronous speech synthesis task that writes audio to S3. The main lesson is not just how to call Polly. It is how to structure a small automation pipeline so each stage has a clear responsibility.
